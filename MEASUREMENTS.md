| สิ่งที่เฉลยบอก | ความจริงจากภาพ |

**หลักฐานว่าเฉลยขัดแย้งในตัวเอง:** เฟรม 02:06 (เฉลย = "ถอด") กับ 02:21 (เฉลย = "ใส่อยู่")
มีสภาพเหมือนกันทุกประการคือหน้ากากห้อยที่คาง

**ผลกระทบต่อตัวเลข**

| | เฉลยเดิม | **เฉลย v4** |
|---|---|---|
| การละเมิดจริงทั้งหมด | 24 จุด | **56 จุด** |
| recall (เฟรมเต็ม) | 16/24 (67%) | 26/56 (46%) |
| FP (เฟรมเต็ม) | **13** | **3** |
| **หน้ากาก + SR (ไม่มี gate)** | **88% แต่ FP=23** | **80% FP=2** |
| **หน้ากาก + SR + face-gate** | **62% FP=4** | **26% FP=0** |

**ผลกระทบที่ร้ายแรงกว่าตัวเลข — เฉลยที่ผิดทำให้ตัดสินใจผิด**

ลำดับเหตุการณ์จริงที่เกิดขึ้นในโครงงานนี้

1. ใช้ super-resolution ขยายภาพศีรษะ → โมเดลตรวจหน้ากากได้ **80% ซึ่งถูกต้อง**
2. แต่เฉลยที่ผิดทำให้เห็นเป็น **"FP พุ่งจาก 9 เป็น 23"**
3. จึงเพิ่ม **face-visibility gate** เข้ามาเพื่อกด FP
4. gate **ทิ้งคำตอบที่ถูกไป 19 จุด (80% → 26%)** เพื่อกำจัด FP ที่ไม่มีอยู่จริง
5. บันทึกไว้ว่า **"แก้สำเร็จ FP เหลือ 4"** และนำ gate เข้าไปในผลงาน

**สรุป: เครื่องมือวัดที่ผิด ทำให้เพิ่มกลไกที่ทำให้ระบบแย่ลง 3 เท่า โดยเข้าใจว่ากำลังทำให้ดีขึ้น**
→ **ข้อเสนอ: ถอด face-gate ออก** และใช้ super-resolution แบบไม่มี gate

### 2.9 ตรวจซ้ำ FP ถุงมือทั้ง 3 จุด — **เฉลยถูก ตัวเลขไม่เปลี่ยน**

หลังพบว่าเฉลยเคยผิดมา 3 รุ่นติด จึงตรวจ FP ถุงมือทั้ง 3 จุดซ้ำ เผื่อเป็นเฉลยผิดอีก
วิธี: crop มือคนที่ 2 ขยาย 3 เท่า (1140×660) **และดูเฟรมข้างเคียง ±8 วินาที** เพื่อดูสถานะตอนผ้าขยับพ้นมือ

| จุด | โมเดลตอบ | เฉลย | หลักฐานจากเฟรมข้างเคียง | สรุป |
|---|---|---|---|---|
| 00:30 | ถอด | ใส่ | 00:26–00:34 เห็นถุงมือขาวทั้งสองมือเกือบทุกเฟรม | ✅ เฉลยถูก — **FP จริง** |
| 01:33 | ถอด | ใส่ | 01:29–01:37 เห็นถุงมือขาวชัด | ✅ เฉลยถูก — **FP จริง** |
| 02:30 | ถอด | ใส่ | 02:24 / 02:32 / 02:36 เห็นถุงมือขาวเต็มมือขณะจับผ้าแดง | ✅ เฉลยถูก — **FP จริง** |

**สรุป: recall ถุงมือคงเดิม 4/7 (57%) · FP คงเดิม 3 · รวมคงเดิม 46/56 (82%)**

> นี่เป็น**การตรวจครั้งแรกที่ยืนยันว่าเฉลยถูก** (สี่ครั้งก่อนหน้าพบว่าเฉลยผิดทุกครั้ง) — เป็นสัญญาณว่าเฉลย v4 เริ่มนิ่ง

**ได้ของสำคัญกว่าตัวเลข: รู้สาเหตุความผิดพลาดของถุงมือครบทั้งสองโหมด**

| โหมด | จำนวน | เกิดที่ | สาเหตุ |
|---|---|---|---|
| **FP** — บอกถอดทั้งที่ใส่ | 3 | 00:30 · 01:33 · 02:30 | **ผ้าเช็ดสีแดงบังมือ/อยู่ติดมือ** → โมเดลไม่เห็นขาว จึงสรุปว่าไม่ได้ใส่ |
| **FN** — บอกใส่ทั้งที่ถอด | 3 | 02:09 · 02:12 · 02:15 | **ถอดข้างเดียว** → โมเดลเห็นมืออีกข้างที่ยังใส่อยู่ จึงสรุปว่าใส่ |
| **TP** | 4 | 03:36 · 03:39 | ถอดทั้งสองข้าง มือเปลือยชัดเจน → ตรวจได้ครบ |

**ตีความ:** ความผิดพลาดทั้ง 6 จุดมาจากรากเดียวกัน — โมเดลตัดสิน "ใส่ถุงมือไหม" ที่ **ระดับคน ไม่ใช่ระดับมือ**
เมื่อมือสองข้างสถานะต่างกัน (ถอดข้างเดียว) หรือมือข้างหนึ่งถูกบัง โมเดลจะยุบเหลือคำตอบเดียว
→ ทางแก้เชิงสถาปัตยกรรมคือ **แยก probe รายมือ** (แนวเดียวกับที่ ROI แก้ปัญหาระบุลัง) ไม่ใช่การแก้ prompt

> **ผ้าแดงคือผ้าเช็ดที่ถืออยู่ ไม่ใช่ถุงมือสีแดง** — ยืนยันที่ 02:09 ซึ่งเห็นมือเปลือยกำผ้าอยู่ชัดเจน

> อ่านให้ถูกคือ *"ในจุดที่ตรวจแล้ว ยังไม่พบข้อผิดพลาด"* ไม่ใช่ *"แม่นยำ 100%"*
>
> **ตรวจซ้ำเมื่อ 2026-07-17:** ยืนยัน 8/8 โดยเทียบ `roi_scan.json` กับเฉลยบนเฟรม `_frames_full`
> ทำซ้ำได้โดยไม่ต้องใช้ GPU เพราะคำตอบทุกวินาทีถูกบันทึกไว้แล้ว
>
> 🔴 **เคยเขียนผิดเป็น "7/7"** ในหัวข้อนี้และในรายงาน ทั้งที่ตัวตรวจมี 4 จุด × 2 ลัง = **8 การตรวจ**
> และหัวข้อ 8.6 ของรายงานเองก็เขียนว่า **"5/8 → 8/8"** = **รายงานขัดแย้งกันเอง** แก้แล้วทุกจุด
>
> ⚠️ **กับดักที่เกือบพลาดซ้ำรอบสอง:** เฉลยใน `roi_probe.py` เขียนไว้สำหรับเฟรมชุด `_frames`
> (แตกด้วย `ffmpeg -ss`) แต่ `roi_scan.py` รันบนเฟรมชุด `_frames_full` (แตกด้วย `fps=1`)
> เอามาเทียบกันตรง ๆ จะได้ **5/8 และสรุปผิดว่าโมเดลแย่**
> หลักฐาน: ที่ t = 4 เฟรมสองชุดต่างจังหวะกันจนเป็นคนละเหตุการณ์ —
> `_frames` มือ**อยู่นอกลัง** ส่วน `_frames_full` มือ**อยู่ในลัง WAIT** ชัดเจน
> **นี่คือกับดักเดียวกับที่บันทึกไว้ในหัวข้อ 8.6 เป๊ะ ๆ**

## 5.2 ⚠️ ขอบเขตเฉลยเหตุการณ์ไม่ครอบคลุมทุกหมวด — ทำให้ความแม่นต่ำกว่าจริง

## 7. คำสั่งที่ใช้ทำซ้ำผลทั้งหมด

```bash
# บนเครื่อง GPU (co-located)
python3 preflight.py                       # ยืนยันว่าเป็น Cosmos-Reason2-8B จริง
python3 image_tier.py                      # PPE eval 74 เฟรม -> image_obs.json (121 วิ)
python3 som_sr_ppe.py                      # หน้ากาก + super-resolution -> image_obs_sr.json
python3 bench_report.py --endpoints <ep0> --label 1gpu
python3 bench_report.py --endpoints <ep0>,<ep1> --label 2gpu
python3 roi_scan.py --endpoints <ep0>,<ep1>   # ROI grounding ทั้งคลิป (67 วิ)

# บนเครื่องไหนก็ได้ (ไม่ใช้ GPU)
python diff_gt.py                          # <-- คำสั่งเดียวได้ตัวเลขทุกตัวในรายงาน
python verify_roi.py                       # ตรวจผลระบุลัง 8/8 (ไม่ต้องใช้ GPU)
python sop_roi.py                          # สรุปเหตุการณ์ระดับลัง
```

**`python diff_gt.py` (ไม่ใส่ argument) = คอนฟิกที่ส่งมอบ** และพ่นตัวเลขที่อ้างในรายงานครบทุกตัว:

```
  รายการ      จับได้/ละเมิดจริง   recall   FP
  หมวก                    14/14    100%    0
  ถุงมือ                    4/7     57%    3
  หน้ากาก                 28/35     80%    2
  --------------------------------------------
  รวม                     46/56     82%    5
```

| คำสั่ง | ได้อะไร | ตรงกับหัวข้อ |
|---|---|---|
| `python diff_gt.py` | หมวก/ถุงมือ จากเฟรมเต็ม **+ หน้ากากจาก SR** = ที่ส่งมอบ | **§2.5** (46/56, 82%, FP 5) |
| `python diff_gt.py --no-sr` | เฟรมเต็มล้วน (หน้ากากไม่ผ่าน SR) | §2.2 (26/56, 46%, FP 3) |

> ⚠️ **เคยเป็นกับดัก:** เดิม `diff_gt.py` รับ `--obs` ไฟล์เดียว ทำให้คำสั่งในหัวข้อนี้พ่น **46%**
> ขณะที่รายงานเขียน **82%** — ใครรันตามเอกสารจะได้คนละเลขและสรุปว่ารายงานโกหก
> สาเหตุคือคอนฟิกที่ส่งมอบใช้ **สองไฟล์** (เฟรมเต็ม + SR) แต่สคริปต์อ่านได้ไฟล์เดียว
> แก้แล้วโดยเพิ่ม `--mask-obs` (ค่าเริ่มต้น `image_obs_sr.json`) ให้ค่า default = คอนฟิกที่ส่งมอบ
> **บทเรียน: "ทำซ้ำได้" ต้องแปลว่ารันคำสั่งในเอกสารแล้วได้เลขในเอกสาร ไม่ใช่แค่มีสคริปต์อยู่**
