# ภาคผนวก ก — คู่มือการติดตั้งอย่างละเอียด

> คู่มือนี้เขียนจากขั้นตอนที่ใช้จริงในโครงงาน ทุกคำสั่งผ่านการรันจริงแล้ว
> รวมถึงข้อผิดพลาดที่เคยเจอและวิธีแก้ เพื่อให้ผู้ติดตั้งไม่ต้องเสียเวลาลองผิดซ้ำ

## ก.2 ขั้นตอนที่ 1 — เตรียมเครื่องและดาวน์โหลดแบบจำลอง

แบบจำลอง Cosmos-Reason2-8B เป็นแบบจำลองที่ต้องขอสิทธิ์เข้าถึงก่อน (gated model)
จึงต้องเข้าสู่ระบบ Hugging Face ด้วยโทเคนส่วนตัวก่อนดาวน์โหลด

```bash
pip install "huggingface_hub[cli]"
hf auth login                      # ใส่โทเคนจาก huggingface.co/settings/tokens
hf download nvidia/Cosmos-Reason2-8B --local-dir /workspace/models
```

> **ข้อควรระวัง:** คำสั่ง `huggingface-cli` แบบเดิมถูกยกเลิกแล้ว ให้ใช้ `hf` แทน
> และ **ควรเพิกถอนโทเคนหลังใช้งานเสร็จ** โดยเฉพาะเมื่อทำงานบนเครื่องเช่า

## ก.3 ขั้นตอนที่ 2 — ตรวจสอบว่าเครื่องว่างจริง

**ขั้นตอนนี้ห้ามข้าม** โดยเฉพาะเมื่อใช้เครื่องเช่า เพราะแม่แบบของผู้ให้บริการมักรันแบบจำลองอื่น
ค้างไว้และจองหน่วยความจำการ์ดจอไว้ทั้งหมด ทำให้ติดตั้งไม่สำเร็จโดยไม่มีข้อความบอกสาเหตุชัดเจน

```bash
nvidia-smi                         # ดูว่ามีโปรแกรมใดจองหน่วยความจำการ์ดจออยู่
supervisorctl stop vllm            # หยุดบริการที่แม่แบบรันค้างไว้ (ถ้ามี)
kill <PID ที่เห็นจาก nvidia-smi>    # ปิดโปรแกรมที่ยังค้าง
nvidia-smi                         # ยืนยันว่าหน่วยความจำว่างแล้วทั้งสองใบ
```

> **ข้อควรระวัง 2 ข้อที่เจอจริง**
> 1. **อย่าใช้ `pgrep` ค้นหาโปรแกรมเพื่อสั่งปิด** เพราะคำสั่งจะจับตัวเองไปด้วย
>    ให้อ่านรายการโปรแกรมจาก `nvidia-smi` โดยตรง
> 2. **แม้ปิดโปรแกรมหลักแล้ว ยังอาจมีโปรแกรมเบื้องหลังค้างและจองหน่วยความจำอยู่**
>    ต้องตรวจ `nvidia-smi` ซ้ำจนแน่ใจว่าว่างจริง

## ก.4 ขั้นตอนที่ 3 — เริ่มระบบให้บริการแบบจำลอง

**กรณีการ์ดจอ 2 ใบ (แนะนำ)** — แต่ละใบรันแบบจำลองครบทั้งตัวแยกกัน ไม่ต้องสื่อสารข้ามการ์ด
ใบแรกใช้พอร์ต 18000 ใบที่สองใช้พอร์ต 18001

```bash
bash serve_cosmos_2gpu.sh          # เริ่มทั้งสองใบ
```

สคริปต์นี้เรียกใช้คำสั่งต่อไปนี้กับการ์ดแต่ละใบ

```bash
CUDA_VISIBLE_DEVICES=<0 หรือ 1> vllm serve nvidia/Cosmos-Reason2-8B \
  --host 127.0.0.1 --port <18000 หรือ 18001> \
  --served-model-name cosmos \
  --download-dir /workspace/models \
  --max-model-len 16384 \
  --gpu-memory-utilization 0.90 \
  --limit-mm-per-prompt '{"video":1,"image":2}' \
  --allowed-local-media-path /workspace \
  --reasoning-parser qwen3
```

**คำอธิบายค่าที่สำคัญ** — `--gpu-memory-utilization 0.90` คือสัดส่วนหน่วยความจำที่ยอมให้ใช้
ตั้งสูงกว่านี้อาจทำให้ระบบหยุดทำงาน · `--limit-mm-per-prompt` กำหนดว่าหนึ่งคำขอส่งภาพได้
ไม่เกิน 2 ภาพ ซึ่งตรงกับที่ระบบใช้จริง (ป้อนภาพ 2 เฟรมติดกัน) · `--reasoning-parser qwen3`
จำเป็นเพราะแบบจำลองตระกูลนี้ตอบโดยแยกส่วนการให้เหตุผลออกจากคำตอบ

**กรณีการ์ดจอ 1 ใบ** ใช้ `bash serve_cosmos.sh` ซึ่งจะเปิดเฉพาะพอร์ต 18000
ระบบยังทำงานได้แต่ความเร็วลดเหลือประมาณ 0.47 เท่าของความเร็ววิดีโอ

## ก.5 ขั้นตอนที่ 4 — ตรวจสอบก่อนใช้งานจริง (ห้ามข้าม)

```bash
python3 preflight.py
```

สคริปต์นี้จะสอบถามระบบว่ากำลังให้บริการแบบจำลองชื่ออะไรอยู่ แล้วเทียบกับที่ต้องการ
**ถ้าไม่ตรงจะหยุดทันทีและไม่ยอมทำงานต่อ**

> **เหตุผลที่ต้องมีขั้นตอนนี้:** ผู้ให้บริการเครื่องเช่าตั้งค่าเริ่มต้นของเครื่องไว้ชี้ไปที่
> แบบจำลองอีกตัวหนึ่ง โปรแกรมที่เขียนว่า "ถ้าเครื่องกำหนดชื่อแบบจำลองไว้แล้วให้ใช้ค่านั้น"
> จะหยิบค่าของผู้ให้บริการไปใช้โดยไม่มีสัญญาณเตือน ทำให้เสี่ยงต่อการใช้แบบจำลองผิดตัว
> โดยไม่รู้ตัว ซึ่ง **ตรวจจับได้ยากมากเพราะแบบจำลองทั้งสองตัวตอบเป็นภาษาไทยได้เหมือนกัน**

## ก.6 ขั้นตอนที่ 5 — เตรียมเฟรมภาพ

ระบบใช้เฟรมภาพ **สองชุดที่แตกด้วยวิธีต่างกัน และห้ามสลับกันเด็ดขาด**

```bash
# ชุดที่ 1 — สำหรับตรวจอุปกรณ์ป้องกัน (image_tier.py แตกให้เองอัตโนมัติ)
ffmpeg -ss <วินาที> -i cam_a05_1080p.mp4 -frames:v 1 t<วินาที>.jpg

# ชุดที่ 2 — สำหรับคำบรรยายและการระบุลัง
ffmpeg -i cam_a05_1080p.mp4 -vf fps=1 -start_number 0 _frames_full/t%04d.jpg
```

> **ข้อควรระวังที่สำคัญที่สุดในคู่มือนี้:** สองวิธีนี้ให้เฟรมที่**ต่างกันเสี้ยววินาที**
> ซึ่งมากพอที่จะเป็นคนละเหตุการณ์ ตัวอย่างที่พบจริงคือที่วินาทีที่ 4 ชุดหนึ่งมือของผู้ปฏิบัติงาน
> อยู่นอกลัง อีกชุดหนึ่งมืออยู่ในลังกำลังหยิบกระป๋อง **การนำสองชุดมาปนกันทำให้สรุปผลผิด**
> · ต้องแตกจากไฟล์ความละเอียด **1080p** เท่านั้น (ไฟล์ 720p ใช้เล่นบนหน้าเว็บอย่างเดียว)
> · ต้องมี `-start_number 0` มิฉะนั้นเลขเฟรมจะเริ่มที่ 1 และเหลื่อมจากเวลาจริงทั้งคลิป

## ก.7 ขั้นตอนที่ 6 — ตั้งค่ากุญแจสำหรับชั้นวิเคราะห์กฎ

ชั้นวิเคราะห์กฎความปลอดภัยเรียกใช้แบบจำลองภาษาผ่านบริการภายนอก จึงต้องมีกุญแจ

```bash
cp .env.example .env               # แล้วแก้ไฟล์ .env ใส่ค่าต่อไปนี้
#   OPENROUTER_API_KEY=<กุญแจของคุณ>
#   OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
#   OPENROUTER_MODEL=anthropic/claude-opus-4.8
```

> **ห้ามบันทึกกุญแจลงในโค้ดหรือส่งขึ้นระบบควบคุมเวอร์ชันเด็ดขาด**
> โครงงานนี้กำหนดให้ไฟล์ `.env` ถูกละเว้นจากระบบควบคุมเวอร์ชันไว้แล้ว

## ก.8 ขั้นตอนที่ 7 — รันระบบและตรวจสอบว่าติดตั้งสำเร็จ

```bash
# --- ต้องมีการ์ดจอ ---
python3 image_tier.py                                    # ตรวจ PPE 74 เฟรม (~121 วินาที)
python3 som_sr_ppe.py                                    # ตรวจหน้ากากแบบขยายภาพ
python3 roi_scan.py --endpoints <ep0>,<ep1>              # ระบุลังทั้งคลิป (~67 วินาที)
python3 generate_timeline.py --no-box-names --endpoints <ep0>,<ep1>   # คำบรรยาย 220 จุด (~311 วินาที)

# --- ไม่ต้องใช้การ์ดจอ ---
python sop_roi.py                  # สรุปกิจกรรมระดับลัง
python compose_narration.py        # เติมชื่อลังจากผลการครอบภาพลงในคำบรรยาย
python analyze_safety.py           # จับคู่กฎ R1–R10 และระบุระดับความรุนแรง
python diff_gt.py                  # ตรวจสอบความถูกต้อง
```

โดยที่ `<ep0>` และ `<ep1>` คือ `http://127.0.0.1:18000/v1/chat/completions`
และ `http://127.0.0.1:18001/v1/chat/completions` ตามลำดับ

**ตรวจสอบว่าติดตั้งสำเร็จ** ด้วยการรัน `python diff_gt.py` ซึ่งต้องได้ผลดังนี้

```
  รายการ      จับได้/ละเมิดจริง   recall   FP
  หมวก                    14/14    100%    0
  ถุงมือ                    4/7     57%    3
  หน้ากาก                 28/35     80%    2
  --------------------------------------------
  รวม                     46/56     82%    5
```

**ถ้าได้ 46% แทนที่จะเป็น 82%** แสดงว่าไฟล์ผลการตรวจหน้ากากแบบขยายภาพหายไป
ให้รัน `python3 som_sr_ppe.py` ใหม่

## ก.9 ตารางสรุปข้อผิดพลาดที่พบบ่อยและวิธีแก้

| อาการ | สาเหตุ | วิธีแก้ |
|---|---|---|
| ระบบให้บริการเริ่มไม่ขึ้น หน่วยความจำการ์ดจอไม่พอ | มีแบบจำลองอื่นค้างอยู่ | ทำตามขั้นตอน ก.3 ให้ครบ แล้วยืนยันด้วย `nvidia-smi` |
| ผลการวัดออกมาแปลก ๆ ไม่ตรงกับที่ควรเป็น | ใช้แบบจำลองผิดตัวโดยไม่รู้ตัว | รัน `python3 preflight.py` ก่อนเสมอ (ขั้นตอน ก.5) |
| ระบบให้บริการหยุดทำงานเมื่อป้อนวิดีโอ | โหมดวิดีโอไม่เสถียร | ใช้โหมดป้อนภาพนิ่ง 2 เฟรมติดกันแทน ซึ่งเป็นค่าเริ่มต้นของระบบอยู่แล้ว |
| `diff_gt.py` ได้ 46% แทน 82% | ไฟล์ผลตรวจหน้ากากแบบขยายภาพหาย | รัน `python3 som_sr_ppe.py` ใหม่ |
| ภาพหลักฐานไม่ตรงกับเวลาในวิดีโอ | ใช้เฟรมผิดชุด | ตรวจว่าใช้เฟรมชุดที่แตกด้วย `-vf fps=1 -start_number 0` (ขั้นตอน ก.6) |
| ชั้นวิเคราะห์กฎไม่ทำงาน | ไม่ได้ตั้งกุญแจ | ตรวจไฟล์ `.env` ตามขั้นตอน ก.7 |
| คำสั่ง `huggingface-cli` ใช้ไม่ได้ | คำสั่งถูกยกเลิกแล้ว | ใช้ `hf download` แทน |

## ก.10 ข้อควรทราบเมื่อใช้เครื่องเช่า

- **เครื่องคิดค่าบริการรายชั่วโมง** จึงควรวางแผนล่วงหน้าว่าจะรันอะไรบ้าง และรันให้ครบในรอบเดียว
- **เครื่องคนละตัวให้ความเร็วต่างกันประมาณ 12%** ทั้งที่สเปกบนกระดาษเหมือนกัน
  เพราะเป็นเครื่องที่แชร์ทรัพยากรกับผู้เช่ารายอื่น หากต้องเปรียบเทียบผลต้องวัดบนเครื่องเดียวกัน
- **ควรวัดผลบนเครื่องเดียวกับที่แบบจำลองทำงาน** ไม่ควรวัดผ่านการเชื่อมต่อระยะไกล
  เพราะเวลาที่ใช้ส่งภาพข้ามเครือข่ายมากกว่าเวลาประมวลผลจริง จนทำให้อ่านผลไม่ออก
- **อย่าลืมปิดเครื่องเมื่อใช้งานเสร็จ** และเพิกถอนโทเคนที่ใช้ดาวน์โหลดแบบจำลอง
