# 5. รายละเอียดของการพัฒนา

## 5.1 เนื้อเรื่องย่อ (Story Board) และผลงานที่พัฒนาขึ้น

### สถานการณ์ที่ระบบเข้าไปแก้

โรงงานติดกล้องวงจรปิดไว้เต็มสายการผลิตอยู่แล้ว แต่กล้องทำได้แค่ "บันทึกไว้" เมื่อเกิดอุบัติเหตุ
เจ้าหน้าที่ความปลอดภัยต้องย้อนดูเทปทีละชั่วโมงเพื่อหาว่าเกิดอะไรขึ้น และในทางปฏิบัติ
แทบไม่มีใครนั่งเฝ้าจอ 8 ชั่วโมงเพื่อจับเหตุการณ์ตอนที่มันกำลังเกิด

ระบบที่พัฒนาขึ้นทำหน้าที่ **อ่านวิดีโอแทนคน แล้วตอบว่า "วินาทีไหนเกิดอะไรขึ้น"**
โดยไม่ต้องเทรนโมเดลใหม่กับกล้องแต่ละตัว

### ฉากที่ใช้พัฒนาและทดสอบ

กล้อง **CAM-A05** จับภาพสายการผลิตกระป๋อง ความยาว **3 นาที 39 วินาที (219 วินาที)** ความละเอียด 720p
ในฉากมีผู้ปฏิบัติงาน 2 คน — คนที่ 1 อยู่ฝั่งซ้าย (หยิบกระป๋องจากลังมาวางบนสายพาน)
คนที่ 2 อยู่ฝั่งขวา (หยิบออกจากสายพาน เช็ด แล้วเก็บลงลัง) มีลังกระดาษ 3 ใบ (WAIT / NG / GOOD)
และสายพานลำเลียงเดินอยู่ตลอด

เจ้าหน้าที่ความปลอดภัยอยากรู้มากที่สุด — ระบบตรวจพบเหตุการณ์นี้ได้

### ผลงานที่พัฒนาขึ้น

ก่อนหน้านี้เส้นเวลาเหตุการณ์ในระบบเป็น **ข้อมูลจำลอง (mock) ที่เขียนขึ้นด้วยมือ**
งานในส่วนนี้คือทำให้ **โมเดลผลิตเส้นเวลานั้นขึ้นมาเองจากวิดีโอจริง**

ผลงานที่ได้มี 3 ชิ้น

1. **เส้นเวลาเหตุการณ์ที่โมเดลสร้างเอง** (`cam_a05_timeline.json`) — แทนที่ข้อมูลจำลองเดิมได้ทั้งหมด
   โดยใช้ schema เดียวกับของเดิม ทำให้ระบบส่วนหน้าไม่ต้องแก้อะไรเลย
2. **หน้าจอสาธิตแบบสด** — วิดีโอเล่นอยู่ฝั่งซ้าย คำเตือนเด้งขึ้นฝั่งขวาตามวินาทีที่โมเดลตรวจพบ
   โดยคำเตือนมาจากโมเดลที่กำลังคิดอยู่ ณ ขณะนั้น ไม่ได้อ่านจากไฟล์
3. **ชุดวัดผล** ที่ใช้เฉลยระดับรายเฟรมเป็นตัวตัดสิน (อธิบายในข้อ 5.5)

## 5.2 ทฤษฎี หลักการ และเทคนิคที่ใช้

### 5.2.1 ทำไมเลือก Vision-Language Model ไม่ใช่ Object Detection

แนวทางที่ใช้กันทั่วไปคือ object detection (เช่น YOLO) ซึ่งให้ความแม่นสูงมาก
แต่มีเงื่อนไขว่า **ต้องเก็บภาพและติดป้ายกำกับ (annotate) แล้วเทรนใหม่ทุกโรงงานและทุกมุมกล้อง**
เมื่อย้ายกล้อง เปลี่ยนแสง หรือเปลี่ยนชุดพนักงาน ก็ต้องเทรนรอบใหม่

งานนี้เลือก **Vision-Language Model (VLM)** ซึ่งเป็นโมเดลที่รับได้ทั้งภาพและข้อความ
และตอบกลับเป็นภาษาธรรมชาติ ข้อได้เปรียบคือ **ปรับพฤติกรรมได้ด้วยการแก้คำสั่ง (prompt)
ไม่ต้องเทรนใหม่** เช่น ถ้าโรงงานเพิ่มกฎ "ต้องสวมแว่นนิรภัย" ก็เพิ่มบรรทัดใน prompt
แล้วใช้ได้ทันที ซึ่งตรงกับข้อจำกัดของโครงงานที่ไม่มีชุดข้อมูลติดป้ายกำกับและไม่มีเวลาเทรน

โมเดลที่เลือกคือ **NVIDIA Cosmos-Reason2-8B** ซึ่งเป็น VLM สาย reasoning
(ผ่านการปรับแต่งมาสำหรับการให้เหตุผลเชิงกายภาพและเชิงพื้นที่)
ได้เปรียบเทียบกับ Qwen3-VL-8B แล้วเลือก Cosmos เพราะให้คำตอบเชิงเหตุผลที่เสถียรกว่าในฉากโรงงาน

### 5.2.2 โมเดลมองภาพเป็นอะไร

ประเด็นที่มักเข้าใจผิดคือ VLM ไม่ได้ "ดูรูป" แบบที่คนดู แต่แปลงรูปเป็นลำดับตัวเลขก่อน
กระบวนการภายในสรุปได้ตามรูปที่ 2

![รูปที่ 2](figures/figure2_gpu_zoom_th.png)

**รูปที่ 2** รายละเอียดการทำงานภายในการ์ดจอหนึ่งใบ ต่องานหนึ่งชิ้น และแผนภูมิเวลาแสดงการแบ่งงานระหว่างการ์ดสองใบ

หลักการสำคัญ 3 ข้อจากรูปที่ 2

**(1) Visual tokenization** — ภาพถูกหั่นเป็นตาราง (patch) ขนาด 14×14 พิกเซล แล้วรวมแบบ 2×2
ผลคือทุกพื้นที่ **28×28 พิกเซล กลายเป็น 1 token** จากนั้นโมเดลอ่าน token เหล่านี้เรียงกัน
เหมือนอ่านคำในประโยค นี่คือเหตุผลโดยตรงที่ **วัตถุเล็กอย่างหน้ากากตรวจยาก** —
เพราะหน้ากากในเฟรม 720p กินพื้นที่ไม่กี่ token เท่านั้น (ดูข้อ 5.5)

**(2) Prefill กับ Decode มีคอขวดคนละแบบ** — ขั้น prefill (อ่าน token ทั้งหมดพร้อมกันด้วย
กลไก attention) เป็นงานที่ติดกำลังคำนวณ (compute-bound) ส่วนขั้น decode (เขียนคำตอบทีละ token)
ติดแบนด์วิดท์หน่วยความจำ (memory-bandwidth-bound) ข้อค้นพบนี้อธิบายผลการทดลองที่ว่า
**การ์ด A100 เร็วกว่า RTX 4090 เพียงประมาณ 1.2 เท่า ไม่ใช่ 2–3 เท่า** ตามที่คาดไว้ตอนแรก
เพราะงานนี้หนักที่ prefill ซึ่ง 4090 มีกำลังคำนวณ fp16 ใกล้เคียง A100 อยู่แล้ว
ข้อสรุปนี้นำไปสู่การเลือกใช้ 4090 สองใบแทน A100 หนึ่งใบ ซึ่งประหยัดกว่าและเร็วกว่า

**(3) เวลาที่ใช้แปรผันตามความยาวคำตอบ** — เขียนเป็นความสัมพันธ์เชิงเส้น

> L(N) ≈ L₀ + N · t_dec   โดยวัดได้ t_dec ≈ 0.017 วินาที/token

วัดจริงบน RTX 4090 สองใบได้ 1.14 วินาทีที่ 60 token และ 1.65 วินาทีที่ 90 token
สมการนี้ใช้ตอบคำถามเชิงออกแบบว่า "ถ้าอยากได้คำบรรยายทุก 1 วินาที ต้องจำกัดคำตอบกี่ token"

### 5.2.3 การขนานงานแบบ Data-Parallel

เมื่อมีการ์ดจอสองใบ ทางเลือกมี 2 แบบ

| แนวทาง | วิธี | ข้อเสีย |
|---|---|---|
| Tensor-parallel | ผ่าโมเดลครึ่งหนึ่งลงแต่ละใบ | ต้องสื่อสารข้ามการ์ดทุกชั้น → มี overhead |
| **Data-parallel (ที่เลือกใช้)** | ใบละสำเนาโมเดลเต็ม สลับกันรับงาน | ใช้ VRAM ซ้ำซ้อน แต่ **ไม่ต้องสื่อสารข้ามการ์ดเลย** |

เนื่องจากโมเดลขนาด 8B ใช้ VRAM ประมาณ 22 GB ซึ่งใส่ลงการ์ด 24 GB ได้พอดี
จึงเลือก data-parallel โดยจ่ายงานแบบ round-robin ตามสูตร

> replica(i) = E(i mod 2)

งานเลขคู่ไปการ์ดใบที่ 1 งานเลขคี่ไปการ์ดใบที่ 2 ทำให้ได้ **ความเร็วเพิ่มขึ้นประมาณ 2 เท่าแบบเชิงเส้น**
ผลคือช่วงเวลาระหว่างคำตอบลดจาก 1.65 เหลือ **0.91 วินาทีต่องาน** ซึ่งเพียงพอสำหรับ
การบรรยายทุก 1 วินาทีขณะวิดีโอเล่นด้วยความเร็วปกติ

### 5.2.4 เทคนิคเชิง Prompt (ไม่มีการเทรน)

| เทคนิค | สิ่งที่ทำ | เหตุผล |
|---|---|---|
| Context grounding | ระบุในคำสั่งว่าคนที่ 1 อยู่ซ้าย คนที่ 2 อยู่ขวา มีลัง WAIT/NG/GOOD | ลดการสับสนตัวบุคคล |
| Frame pairing | ส่ง 2 เฟรมติดกัน (t, t+1) ต่อหนึ่งคำถาม | เฟรมเดียวไม่มีข้อมูลการเคลื่อนไหว |
| Abstention | อนุญาตให้ตอบว่า "มองไม่เห็น" ได้ | กันโมเดลเดาเมื่อมือถูกบัง |
| Chain-of-thought | ให้อธิบายเหตุผลก่อนสรุป | Cosmos ตอบใน field `reasoning` |

### 5.2.5 การกรองสัญญาณรบกวนด้วย State Machine

ระหว่างทดสอบพบว่าที่วินาที 01:08 ระบบแจ้งว่า "ผู้ปฏิบัติงานคนที่ 1 ไม่สวมถุงมือ"
ทั้งที่ในภาพจริงสวมอยู่ปกติ เมื่อเปิดเฟรมนั้นดูพบว่า **มือถูกลดลงต่ำจนมองไม่เห็นในเฟรม**
โมเดลจึงเดาว่า "ไม่มี"

วิธีแก้ไม่ได้อยู่ที่ prompt แต่อยู่ที่ **การประมวลผลภายหลัง** โดยสร้าง state machine
เชิงเวลา (`ppe_tracker.py`) ที่มีกฎ 2 ข้อ

1. **มองไม่เห็น → คงสถานะเดิมไว้** ไม่นับเป็นการละเมิด
2. **ต้องเห็นซ้ำ 2 ครั้งติดกัน** จึงยืนยันว่าละเมิดจริง (debounce)

หลังแก้ วินาที 01:08 ไม่แจ้งเตือนอีก ขณะที่การละเมิดจริงที่ 01:36 และ 03:00 ยังตรวจพบครบ
บทเรียนเชิงวิศวกรรมจากจุดนี้คือ **ความแม่นยำแก้ที่ post-processing ได้ผลกว่าการไล่แก้ prompt**

## 5.3 เครื่องมือที่ใช้ในการพัฒนา

### ภาษาและไลบรารีหลัก

| หมวด | เครื่องมือ | ใช้ทำอะไร |
|---|---|---|
| ภาษา | **Python 3** | ทั้งระบบ |
| Inference server | **vLLM** | เสิร์ฟโมเดล Cosmos ผ่าน OpenAI-compatible API |
| โมเดลหลัก | **NVIDIA Cosmos-Reason2-8B** (BF16) | ดูภาพ + ให้เหตุผล |
| Deep learning | **PyTorch** | รากฐานของ vLLM |
| เรียก API | **requests** | คุยกับ vLLM และ OpenRouter |
| ประมวลผลภาพ | **Pillow (PIL)** | ครอบภาพ ทำ montage สำหรับเฉลย |
| Super-resolution | **super-image (EDSR-4x)** | ขยายภาพศีรษะเพื่อตรวจหน้ากาก |
| ตัด/ย่อวิดีโอ | **FFmpeg** (ผ่าน `imageio-ffmpeg`) | ดึงเฟรม 1 fps, ย่อเป็น 720p |
| ขนานงาน | **concurrent.futures** (stdlib) | ThreadPool จ่ายงานลง 2 GPU |
| เว็บสาธิต | **http.server** (stdlib) | เสิร์ฟหน้าเว็บ + สตรีมวิดีโอแบบ range request |
| วิเคราะห์ข้อความ | **Claude Opus** ผ่าน OpenRouter | จับคู่กฎความปลอดภัย + สรุป SOP |

**ข้อสังเกตด้านการออกแบบ:** หน้าเว็บสาธิตใช้ `http.server` จากไลบรารีมาตรฐาน
ไม่ได้ใช้เฟรมเวิร์กอย่าง Flask/FastAPI เพราะความต้องการมีแค่ 3 endpoint
และต้องการให้ติดตั้งบนเครื่องเช่า GPU ได้โดยไม่ต้องลง dependency เพิ่ม

### โครงสร้างพื้นฐาน

- **Vast.ai** — เช่าเครื่อง GPU รายชั่วโมง (RTX 3090 / RTX 4090 / A100 สำหรับเปรียบเทียบ)
- **Hugging Face Hub** — ดาวน์โหลดน้ำหนักโมเดล
- **SSH port forwarding** — ต่อจากเครื่องพัฒนาเข้าเครื่อง GPU

### เครื่องมือที่ทดลองแล้วไม่ได้ใช้ในผลงานสุดท้าย

| เครื่องมือ | ผลการทดลอง |
|---|---|
| **Ultralytics YOLO / YOLO-World** (`ppe_yolo_probe.py`, `yolow_probe.py`) | โมเดล PPE สำเร็จรูปย้ายมาใช้กับฉากนี้ไม่ได้ (transfer fail) และการเทรนใหม่ขัดข้อจำกัดโครงงาน |
| **Grounding DINO** (`gd_probe.py`) | ทดลองใช้ชี้ตำแหน่งวัตถุแบบ zero-shot |
| **Set-of-Mark prompting** (`som_*.py`) | ครอบภาพแล้วติดหมายเลขให้ VLM — ช่วยเรื่องหน้ากาก แต่**ทำให้การตรวจถุงมือแย่ลง** จึงไม่ใช้ |

การบันทึกสิ่งที่ทดลองแล้วไม่ผ่านไว้ด้วย เป็นไปเพื่อให้ผู้พัฒนาต่อไม่ต้องเสียเวลาลองซ้ำ

## 5.4 รายละเอียดเชิงเทคนิคของซอฟต์แวร์ (Software Specification)

### 5.4.1 โครงสร้างของซอฟต์แวร์ (Design)

![รูปที่ 1](figures/figure1_overview_th.png)

**รูปที่ 1** ภาพรวมสถาปัตยกรรม แสดงการไหลของข้อมูลตั้งแต่วิดีโอเข้าจนถึงหน้าจอ

ระบบแบ่งเป็น 6 ชั้น โดยมี **หลักคิดหลักคือแยก "จับให้แม่น" ออกจาก "เข้าใจให้ลึก"**
เพราะสองอย่างนี้มีธรรมชาติต่างกัน — การแจ้งเตือนพลาดไม่ได้ ส่วนการบรรยายบริบทต้องเล่าได้รื่น

| ชั้น | หน้าที่ | ทรัพยากร |
|---|---|---|
| L0 ภาพเข้า | รับวิดีโอจากกล้อง | — |
| L1 เตรียม | ตัดเฟรม 1 fps → 220 เฟรม, จับคู่ (t, t+1), เข้ารหัส base64 | CPU |
| **L2 มองเห็น** | **VLM อ่านภาพ เขียนคำบรรยาย** | **GPU × 2** |
| L3 ประกอบ | รวมเป็น 2 แทร็ก: เหตุการณ์เสี่ยง + คำบรรยายรายวินาที | CPU |
| L4 วิเคราะห์ | จับคู่กฎความปลอดภัย R1–R10 → ระดับความรุนแรง, ตรวจ SOP | LLM (ไม่ใช้ GPU) |
| L5 ผลลัพธ์ | แสดงบนหน้าจอ (โหมดสด / โหมดเล่นย้อนหลัง) | — |

**มีเพียงชั้น L2 ชั้นเดียวที่ต้องใช้การ์ดจอ** ซึ่งเป็นการออกแบบที่ตั้งใจ —
เมื่อผลิตไฟล์เส้นเวลาเสร็จแล้ว การสาธิตสามารถเล่นย้อนหลังได้โดยไม่ต้องใช้ GPU เลย
ทำให้ต้นทุนการสาธิตเป็นศูนย์

### 5.4.2 สถาปัตยกรรม 2 ระดับ (2-Tier) — และหลักฐานว่าจำเป็น

ในชั้น L2 งานถูกแยกเป็น 2 ระดับ ซึ่ง **ไม่ได้ออกแบบไว้แต่แรก แต่เกิดจากการทดลองที่ล้มเหลว**

| ระดับ | ใช้ตรวจ | วิธี | เวลา |
|---|---|---|---|
| **Image tier** | PPE, คนนอก, ของแปลกปลอม | เฟรมเดียว 1080p | ~1.85 วินาที |
| **Video tier** | SOP, near-miss | หลายเฟรมต่อเนื่อง 480p | ~15–30 วินาที |

เหตุผลที่ต้องแยก

- **โหมดวิดีโอย่อภาพลงมาก → มองไม่เห็น PPE** (ตรวจได้ 0 จาก 2 รายการ)
- **เฟรมเดียวตัดสิน SOP และ near-miss ไม่ได้** เพราะทั้งคู่เป็นเรื่องของการเคลื่อนไหว
  ต้องมีข้อมูลเชิงเวลา

## 5.5 ขอบเขตและข้อจำกัดของโปรแกรม

ส่วนนี้รายงานสิ่งที่ **ทำไม่ได้** อย่างตรงไปตรงมา พร้อมสาเหตุที่ตรวจสอบแล้ว

### 5.5.1 วิธีที่ใช้วัด — การสร้างเฉลยอ้างอิง

ปัญหาที่พบระหว่างพัฒนาคือ **ตัวเลขความแม่นยำที่วัดตอนแรกเชื่อไม่ได้** เพราะเฉลยชุดแรก
ตัดสินจากภาพรวมทั้งเฟรม จึงมองการละเมิดที่สังเกตยากไม่ออก เช่น ถอดถุงมือข้างเดียว
หรือหน้ากากที่ดึงลงคาง ทำให้การละเมิดบางช่วงหลุด

จึงสร้างเฉลยใหม่โดยครอบภาพผู้ปฏิบัติงานแต่ละคนด้วยความละเอียดสูง แล้วตรวจทีละจุด
ได้ **74 จุดตรวจ (ทุก 3 วินาที ตลอด 219 วินาที)** เป็น `claude_gt.json`
ผลคือ **ตัวเลข recall ตกจาก 81% เหลือ 71%** ซึ่งเป็นตัวเลขที่ถูกต้องกว่า —
การวัดที่หลวมทำให้ระบบดูดีเกินจริง

> **ข้อจำกัดที่ยังเหลือของเฉลยชุดนี้:** 74 จุด = **34% ของ 220 เฟรม** ยังไม่ได้ตรวจครบทุกเฟรม
> การละเมิดที่เกิดและจบภายในช่วง 3 วินาทีระหว่างจุดตรวจอาจหลุดจากการวัด —
> การขยายเฉลยให้ครบทุกเฟรมเป็นงานที่ควรทำต่อ

### 5.5.2 ผลการวัดจริง

วัดจาก 74 เฟรม พบการละเมิด PPE จริง 24 จุด

| ตัวชี้วัด | ผล |
|---|---|
| ตรวจพบ | **17 จาก 24 (71%)** |
| **แจ้งเตือนผิด (False Positive)** | **0** |
| ตรวจไม่พบ (False Negative) | 7 — **เป็นหน้ากากทั้งหมด** |

แยกรายอุปกรณ์

| อุปกรณ์ | ตรวจพบ | หมายเหตุ |
|---|---|---|
| หมวก | **100%** | ไม่มีแจ้งผิดเลย |
| ถุงมือ | **100%** | |
| **หน้ากาก** | **12% (1 จาก 8)** | ใช้เฟรมเต็ม |
| หน้ากาก + Super-resolution | **62%** | EDSR-4x + face gate, ทำได้เฉพาะแบบออฟไลน์ |

**การที่ False Positive เป็นศูนย์มีความหมายในทางปฏิบัติ** — ระบบความปลอดภัยที่แจ้งผิดบ่อย
จะถูกเจ้าหน้าที่ปิดทิ้งภายในไม่กี่วัน ระบบนี้เลือกพลาดบางอย่างดีกว่าแจ้งผิด

### 5.5.3 ข้อจำกัดที่ 1 — หน้ากาก คือเพดานของการรับรู้

หน้ากากที่ถูกดึงลงมาไว้ที่คางมีขนาดเล็กมากในเฟรม และมีสัญญาณรบกวนจากการบีบอัดวิดีโอ
เมื่อคำนวณตามหลักการในข้อ 5.2.2 (28×28 พิกเซล = 1 token) พื้นที่หน้ากากเหลือเพียงไม่กี่ token
โมเดลจึงรายงานเองว่า `face_visible = false` แม้จะขยายภาพแล้ว

ได้ตรวจสอบด้วย 3 แนวทางและยืนยันว่าเป็นเพดานจริง

| แนวทางที่ลอง | ผล |
|---|---|
| ใช้ detector หน้ากากสำเร็จรูป | ย้ายมาใช้กับฉากนี้ไม่ได้ |
| Set-of-Mark (ครอบภาพ + ติดหมายเลข) | หน้ากากดีขึ้น แต่**ถุงมือแย่ลง** |
| Super-resolution EDSR-4x | ได้ 62% แต่ใช้เวลา ~3 วินาที/ศีรษะบน CPU |

**บทเรียนสำคัญ:** ตอนแรกเกือบรับ Set-of-Mark มาใช้เพราะเห็นว่าหน้ากากดีขึ้น
แต่เมื่อวัด **ทุกรายการก่อน–หลัง** จึงพบว่ามันทำให้ถุงมือพัง →
**การวัดต้องดูภาพรวมทั้งหมด ไม่ใช่ดูเฉพาะตัวที่กำลังแก้**

ผลจากข้อจำกัดนี้คือ หน้ากากถูกถอดออกจากเส้นทางการตรวจแบบสด
และแสดงเป็นชั้นข้อมูลที่คำนวณล่วงหน้าแทน (เพราะ super-resolution ช้าเกินไปสำหรับงานสด)

### 5.5.4 ข้อจำกัดที่ 2 — การข้ามขั้นตอน SOP ทำไม่ได้ (เพดานของข้อมูล)

ระบบเคยรายงานว่ามีการ "หยิบของเสียจากลัง NG กลับขึ้นสายพาน" ที่ 01:53 และ 03:27
แต่เมื่อเปิดเฟรมจริงตรวจสอบพบว่า **ผู้ปฏิบัติงานหยิบจากลัง WAIT ซึ่งเป็นงานปกติ**

สาเหตุคือ ลัง NG / WAIT / GOOD **เป็นลังกระดาษหน้าตาเหมือนกันทุกใบ** กระป๋องข้างในก็เหมือนกัน
และมุมกล้องกดลงจากด้านบนทำให้อ่านป้ายไม่ชัด → โมเดลระบุลังผิด

จุดที่ทำให้เรื่องนี้เป็นข้อค้นพบไม่ใช่แค่ข้อบกพร่อง คือ **กฎ SOP ที่มีมูลค่าจริงทุกข้อ
(R2, R7, R8) ล้วนต้องอาศัยการแยกลังทั้งสิ้น** ซึ่งตรงกับจุดที่ VLM ทำไม่ได้พอดี

| ขั้นตอน SOP | ต้องดูอะไร | ทำได้ |
|---|---|:---:|
| 3 วางลงสายพาน · 5 หยิบจากสายพาน · 6 เช็ด | การกระทำที่เห็นได้ | ✅ |
| 1, 2, 7 | แยกลัง WAIT/NG/GOOD | ❌ |
| 2 ตรวจด้วยสายตา · 4 ตรวจด้วยเครื่อง | ความคิด / การทำงานของเครื่อง | ❌ |

**การตัดสินใจ:** ไม่นำฟังก์ชันนี้ออกสู่ผลงาน เพื่อไม่ให้เกิดการแจ้งเตือนผิด
และรายงานไว้เป็นข้อค้นพบพร้อมแนวทางแก้ในอนาคตแทน (ดูข้อ 5.5.6)

|---|---|
| **ทดสอบกับวิดีโอเดียว (N = 1)** | เป็นการจัดฉากถ่ายทำ 1 คลิป การสรุปว่าใช้ได้กับกล้องอื่นยังทำไม่ได้ อยู่นอกขอบเขต |
| **ความละเอียดของเหตุการณ์** | ผลลัพธ์เป็น "ช่วงเหตุการณ์" ไม่ใช่รายวินาทีแบบข้อมูลจำลองเดิม เพราะระดับ video tier ตรวจทุก 15 วินาที การแตกเป็นรายวินาทีจะเป็นการปั้นตัวเลข |
| **ต้องเป็นวิดีโอที่บันทึกไว้แล้ว** | ยังไม่รองรับสตรีม RTSP จากกล้องโดยตรง |
| **โหมดวิดีโอทำให้ vLLM ล่ม** | เกิด `EngineDeadError` จึงต้องใช้โหมดภาพ 2 เฟรมแทน |
| **การแยก near-miss กับการวางปกติ** | ยังกำกวมในบางท่าทาง |
| **ภาษา** | คำบรรยายเป็นภาษาไทยเท่านั้น |

### 5.5.6 แนวทางพัฒนาต่อ

การแก้ปัญหาการแยกลังไม่สามารถทำได้ด้วยการแก้ prompt เพราะเป็นข้อจำกัดของข้อมูลเอง
ต้องแก้ที่ต้นทาง

1. **กำหนดพื้นที่คงที่ (ROI)** — ลังไม่ขยับ จึงกำหนดพิกัดล่วงหน้าได้ว่าโซนใดคือ NG / WAIT / GOOD
   แล้วตรวจว่ามือเข้าโซนใด (แก้ปัญหาได้ตรงที่สุด ต้นทุนต่ำสุด)
2. **ติดป้ายสีหรือ QR ที่ลัง** ให้ต่างกันชัดเจน
3. **ติดตามกระป๋องรายชิ้น (object tracking)**

ทั้งสามแนวทางเป็นงานวิศวกรรมภาพเพิ่มเติม ไม่ใช่การปรับคำสั่ง

## 5.6 คุณลักษณะของอุปกรณ์ที่ใช้กับโปรแกรม

### ความต้องการขั้นต่ำ

| ส่วนประกอบ | ข้อกำหนด | เหตุผล |
|---|---|---|
| GPU | **24 GB VRAM ขึ้นไป** | โมเดล 8B แบบ BF16 ใช้ ~22 GB |
| จำนวน GPU | 1 ใบ (ออฟไลน์) / **2 ใบ (สด)** | 2 ใบเพื่อให้ได้ทุก 1 วินาที |
| ระบบปฏิบัติการ | Linux + CUDA | vLLM ไม่รองรับ Windows |
| พื้นที่ดิสก์ | ~20 GB | น้ำหนักโมเดล + เฟรม |

### ผลการเปรียบเทียบฮาร์ดแวร์ (วัดจริง)

| การ์ด | เวลา 1 คำขอ | ช่วงเวลาระหว่างคำตอบ | ความเร็ววิดีโอ |
|---|---|---|---|
| RTX 3090 × 1 | 5.14 วิ | 5.07 วิ | 0.2× |
| RTX 4090 × 1 | 3.14 วิ | 3.46 วิ | 0.29× |
| A100-40GB × 1 | 2.78 วิ | 2.97 วิ | 0.34× |
| **RTX 4090 × 2** | **1.14 วิ** | **0.91 วิ** | **1.0× (เต็มสปีด)** |

> หมายเหตุ: แถวการ์ดใบเดียววัดผ่าน SSH tunnel จึงมีเวลาเครือข่ายปนอยู่
> ใช้เปรียบเทียบเชิงสัมพัทธ์เท่านั้น แถว 4090 × 2 วัดบนเครื่องเดียวกับโมเดล

### ข้อค้นพบด้านฮาร์ดแวร์

**A100 ไม่คุ้ม** — เร็วกว่า RTX 4090 เพียง ~1.2 เท่า ทั้งที่ราคาสูงกว่ามาก
สาเหตุอธิบายด้วยหลักการในข้อ 5.2.2 คือ งานนี้หนักที่ prefill ซึ่งเป็น compute-bound
และ 4090 มีกำลัง fp16 ใกล้เคียง A100 ส่วนแบนด์วิดท์ที่เหนือกว่าของ A100 ช่วยได้เฉพาะ
ขั้น decode ซึ่งสั้นมากในงานนี้ → **RTX 4090 สองใบ (~$0.70/ชั่วโมง) คุ้มที่สุด**

**เครือข่ายเป็นคอขวด ไม่ใช่ GPU** — ตอนแรกรันหน้าจอสาธิตบนเครื่องพัฒนาแล้วส่งภาพ
ข้าม SSH tunnel ไปยังเครื่อง GPU ได้เวลา 4–5 วินาทีต่อคำขอ เมื่อย้ายไปรันบนเครื่อง GPU
เหลือ **1.65 วินาที** — ส่วนต่างประมาณ 3 วินาทีคือเวลาเครือข่ายล้วนจากการส่งภาพ 1 MB ต่อคำขอ

> **สถาปัตยกรรมที่ถูกต้อง:** รันทุกอย่างบนเครื่อง GPU แล้ว forward เฉพาะหน้าเว็บ (พอร์ต 5003)
> ไม่ส่งภาพข้ามเครือข่าย

### ความต้องการฝั่งผู้ใช้งาน

หากใช้โหมดเล่นย้อนหลัง (อ่านจากไฟล์เส้นเวลา) **ไม่ต้องใช้ GPU เลย**
ต้องการเพียงเว็บเบราว์เซอร์ที่เล่นวิดีโอ MP4 ได้ ซึ่งเป็นโหมดที่แนะนำสำหรับการสาธิตทั่วไป
