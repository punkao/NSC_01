```
ฝั่ง model อัปเดตใหญ่แล้ว pull ล่าสุดก่อน แล้วอ่าน spike/INTEGRATION_FOR_FRONTEND.md
โดยเฉพาะ "ข้อ 9" (ข้อมูล SOP ระดับลัง) กับ "ข้อ 2" (คำสั่งแตกเฟรม) ให้ครบก่อนเริ่มแก้ UI

## สรุปสิ่งที่เปลี่ยน (ทำไมต้องแก้ UI)

เจอบั๊กใหญ่: VLM (Cosmos) ระบุชื่อลังผิด 146/220 จุด (66%) — มันบอกว่า "หยิบจากลัง NG"
ทั้งที่จริงหยิบจากลัง WAIT (เจ้าของวิดีโอยืนยัน + ไล่ดูเฟรมครบ 220 เฟรมแล้ว ไม่มีการหยิบจาก NG เลย)

สาเหตุ (วัดจากภาพจริง): attribute-binding failure ของ VLM
- ลัง WAIT กับ NG วางชิดกันแค่ 251px และเล็ก (4.5% ของภาพ) -> โมเดลจับป้ายผิดลัง
- ลัง GOOD อยู่เดี่ยว ห่าง 1053px -> โมเดลไม่เคยเรียกผิดเลย = ไม่ใช่ "อ่านป้ายไม่ออก" แต่ "จับคู่ผิด"
- ลองแก้ 4 วิธี (layout hint / สั่งห้าม / บอกความรู้โดเมน / ถาม yes-no) -> prompt คุมไม่ได้ 100%

วิธีแก้ที่ใช้ (late fusion — เอางาน "ระบุลัง" ออกจาก VLM):
- VLM ทำแค่: บรรยายการกระทำ + ฝั่งซ้าย/ขวา (สิ่งที่มันแม่น ไม่เคยผิด)
- โค้ดทำ: crop ROI ทีละลัง (กล้องนิ่ง ลังอยู่กับที่) แล้วถาม VLM แค่ "มีมือในกรอบนี้ไหม" (yes/no)
  -> "ลังไหน" มาจากโค้ด 100% ; validate แล้วตรง 7/7 กับการไล่ดูเฟรมด้วยคน (false positive = 0)
- แล้วประกอบ: "หยิบกระป๋องจาก[ลังฝั่งซ้าย]" + ROI บอก WAIT=true -> "หยิบกระป๋องจากลัง WAIT"

## สิ่งที่ต้องทำใน frontend

1. narration ตอนนี้บอกชื่อลังถูกต้องแล้ว (164/220 จุดมีชื่อลังจริง) -> ใช้ field `narration` ตามเดิม ไม่ต้องแก้อะไร
2. เพิ่ม UI รองรับ field ใหม่ 3 ตัว:
   - `box_activity`  : { "<วินาที>": { "WAIT": bool, "NG": bool } }  -> ทำ timeline bar ว่ามือยุ่งลังไหน
   - `sop_events`    : เหตุการณ์มือยุ่งกับลัง (32 รายการ) แต่ละอันมี needs_review + evidence_frame
   - `sop_findings`  : สรุป (wait_interactions=28, ng_interactions=4, ng_review_times=[...])
3. ทำ panel "จุดที่ต้องให้คนตรวจสอบ" — โชว์ sop_events ที่ needs_review=true (มี 4 จุด: 00:10, 01:46, 03:04, 03:28)
   แต่ละจุดโชว์ evidence_frame ให้กดดูได้ + ปุ่ม "ยืนยัน/ไม่ผิด" (คนตัดสิน ไม่ใช่ AI)
4. `sop_compliance` (ของเก่า) ถูกยกเลิกแล้ว ถ้า UI อ้างอยู่ให้เอาออก -> ใช้ `sop_overview` + `sop_events` แทน

## กฎเหล็ก (ห้ามให้ UI เคลมเกินข้อมูล — สำคัญมาก ใช้ตอบกรรมการ)

ทำได้:
- "ระบบตรวจพบว่ามีมือไปยุ่งกับลัง NG 4 ครั้ง — กรุณาตรวจสอบ"

ห้ามทำ:

  เงาคน/กล้องปรับแสงทำค่าแกว่งแรงกว่าสัญญาณจริง 3 เท่า + คลิปมีรอยตัดต่อที่ ~02:17)
- ขึ้น alert R2 อัตโนมัติ -> R2 ต้องให้คนยืนยันเสมอ
- ดึงชื่อลังจากข้อความ narration ไปทำ logic -> ให้ใช้ `box_activity` แทน

## ตรวจสอบย้อนหลังได้ (เผื่อกรรมการถาม "AI รู้เอง หรือใส่ให้?")
ทุก entry ใน timeline มี:
- `narration_vlm` : คำพูดดิบของ Cosmos (ก่อนประกอบ)
- `narration`     : ที่ demo โชว์ (ประกอบชื่อลังแล้ว)

                    "left:ambiguous" = มือคร่อม 2 ลัง ระบบไม่เดา | "left:no_hand" = ไม่เห็นมือในลัง

## ⚠️ เรื่องเฟรม (เจ็บมาแล้ว 2 รอบ — ห้ามพลาด)
เฟรมไม่ได้อยู่ใน git ต้องแตกเอง และ**ต้องใช้คำสั่งนี้เป๊ะ**:

เช็ค: ls _frames_full | wc -l ต้องได้ 220 (t0000.jpg ถึง t0219.jpg)

## ข้อจำกัดที่ยังมี (เขียนในสไลด์ได้เลย = จุดแข็ง ไม่ใช่จุดอ่อน)
```

## 📎 หมายเหตุสำหรับเรา (ไม่ต้องส่งให้เพื่อน)

ไฟล์ที่เกี่ยวกับการแก้รอบนี้:
| ไฟล์ | ทำอะไร |
|---|---|
| `roi_scan.py` | สแกน ROI ทั้งคลิป (ต้องใช้ GPU, 67 วิ) -> `roi_scan.json` |
| `sop_roi.py` | แปลง scan -> `box_activity`/`sop_events`/`sop_findings` (ไม่ใช้ GPU) |
| `compose_narration.py` | ประกอบชื่อลังจาก ROI ลง narration (ไม่ใช้ GPU) |
| `generate_timeline.py` | `--no-box-names` = prompt ห้ามเอ่ยชื่อลัง + `_strip_box_names` |
| `live_demo_combined.py` | live: import prompt/filter/compose/ROI จากไฟล์ข้างบน = ตรงกับ offline เสมอ |

ลำดับรันใหม่ทั้งหมด (ถ้าเปลี่ยนวิดีโอ):
```
1. (GPU) generate_timeline.py --no-box-names --endpoints <ep0>,<ep1>   # ~311s
2. (GPU) roi_scan.py --endpoints <ep0>,<ep1>                           # ~67s
3. sop_roi.py                                                          # local
4. compose_narration.py                                                # local
5. analyze_safety.py                                                   # local (OpenRouter)
6. sop_overview.py                                                     # local (OpenRouter)
```
